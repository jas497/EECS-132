Programming Project 3 Grading Rubric

1. JUnit Testing: 20 points

PointsMetric
20 The logic of the program is correctly broken into small units, and JUnit tests are written for each unit. The tests capture the situations that can occur (test 0, test 1, test many, test middle, test edges). Report correctly describes how the game is broken into units and what should be tested in each unit.
18 Similar to 20 except that a few boundary cases that are likely to cause errors were not tested.
15 JUnit tests were written for several units of the game, the JUnit tests tested most of the necessary cases, and a good report describing the units was provided, but there were several places where multiple units of the game were combined into a single longer method, and thus each separate unit of the method could not be individually tested.
12 At least two reasonable JUnit tests for the game were completed and a testing report describing the units of the game was supplied.
10 At least one reasonable JUnit test for the game was completed.
0 No reasonable JUnit testing done.

2. Readability: 20 points

Basic program structure

PointsMetric
10 Correct comments at top of class, above each method, above each field (if any fields are present), above each loop, and next to or above each variable. (Comments are -not- required next to any index variables used in the loops.) The comments are at most one to three sentences and are a good description. The comments above each loop indicate how the loop works and any prerequisites for the loop. (The loop subgoal/invariant would be nice but is -not- required.) All code follows the proper indentation and naming style. An empty line is placed between the end of one method and the comment of the next.
9 Mostly indented well (up to two places not indented correctly) AND most names correctly follow the naming convention (up to two poorly named variables or methods) AND at most one missing comment on a loop or other complicated code. Each comment is a good description.
8 Good indentation for at least three of the methods AND at least half of the methods and loops have comments that are at most 1-3 sentences and not direct copies of the homework description.
6 At least one method is well commented or there is a comment above every method and the class AND at least one method is nicely indented.
3 At least one method is well commented OR at least one method is nicely indented.
0 No methods have either good comments or good indentation.

JavaDoc Comments

PointsMetric
10All comments above the class, fields, and methods follow the JavaDoc comment style. Correct tags are added for the author, all method parameters and return types.
9All comments above the class, fields, and methods follow the JavaDoc comment style, and most of the comments contain and have correct all required tags. (At most 3 missing or misspelled tags.)
7Most comments above the class, fields, and methods follow the JavaDoc comment style, but a significant number of places are missing tags or the tags are misspelled so the comments are not formatted properly on the webpage.
5Most comments in JavaDoc style of /** but missing most tags OR has most tags but the comments do not start with /** and so do not appear on the webpage.
0No JavaDoc comments provided.

3. Correctness: 60 points

Performance correctness: 40 points

PointsMetric
40 The game runs correctly.
36 The game runs correctly except for one of the possible errors: fails to correctly identify a button click in some situations OR allows swapping of non-adjacent pieces OR fails to mark the correct pieces OR incorrectly marks pieces it should not OR fails to remove pieces after marking them OR fails to slide down pieces to replace the removed pieces OR fails to replace the empty spots at the top of the game with random pieces OR fails to detect a win
32 The game fails on more than one above condition, but most of the time the game correctly swaps adjacent pieces, marks same colored pieces, removes the marked pieces, and slides down the pieces above the removed pieces.

28 In most situations, the game correctly swaps adjacent pieces and marks the appropriate same-colored pieces.
24 The program displays a board of buttons, and at least some of the time, the game correctly swaps adjacent pieces.
16 The program displays a board of buttons.
8 The program at least compiles and does something.
0 No working methods or constructors.

Main method: 10 points

PointsMetric
10A good main method that correctly reads the command line arguments to determine the size of the board and number of jewels and does something appropriate given incorrect command line arguments.
8A good main method that correctly reads the command line arguments and launches the game correctly, but does not handle error conditions appropriately.
6The main method correctly launches the basic game, but it does not correctly launch different board sizes from the command line arguments.
0No main method.

Good code design: 10 points

PointsMetric
10The logic for playing the game is relatively simple and easy for the reader to follow. The names of methods precisely match what the methods do. There is no significant duplicated code.
9The logic for playing the game is relatively simple and easy for the reader to follow. The names of the methods could be more descriptive of their purpose and/or some code parts are duplicates and could be combined.
7The loops are much more complicated than needed and the logic is challenging for the reader to understand.
5All the code is in one method and it is challenging for the reader to understand the logic.
0All the code is in one method and very poorly organized.
