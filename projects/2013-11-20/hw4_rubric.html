Programming Project 4 Grading Rubric

1. JUnit Testing: 20 points

PointsMetric
20 Correct JUnit tests for all methods (inherited, not overridden methods do not need to be re-tested). Tests of conditionals cover the different execution paths. Tests of loops correctly test 0, 1, many and first, middle, last. Report clearly describes the kind of tests needed, and matches JUnit tests written. Descriptions of required tests for incomplete methods and descriptions of code errors are included.
18 Similar to 20 except that a few obvious cases were not tested.
15 Correct JUnit tests for all methods, possibly missing just a few obvious cases, but no testing report provided OR a good testing report provided as well as mostly complete tests for at least three classes including the EmployeeDatabase class.
12 At least once class has all necessary JUnit tests completed.
10 At least a few methods that have conditional statements or loops have correct JUnit tests written.
0 No reasonable JUnit testing done.

2. Readability: 20 points

Basic program structure

PointsMetric
10 Correct comments at top of class, above each method, above each field (if any fields are present), above each loop, and next to or above each variable. (Comments are -not- required next to any index variables used in the loops.) The comments are at most one to three sentences and are a good description. The comments above each loop indicate how the loop works and any prerequisites for the loop. (The loop subgoal/invariant would be nice but is -not- required.) All code follows the proper indentation and naming style. An empty line is placed between the end of one method and the comment of the next.
9 Mostly indented well (up to two places not indented correctly) AND most names correctly follow the naming convention (up to two poorly named variables or methods) AND at most one missing comment on a loop or other complicated code. Each comment is a good description.
8 Good indentation for at least three of the methods AND at least half of the methods and loops have comments that are at most 1-3 sentences and not direct copies of the homework description.
6 At least one method is well commented or there is a comment above every method and the class AND at least one method is nicely indented.
3 At least one method is well commented OR at least one method is nicely indented.
0 No methods have either good comments or good indentation.

JavaDoc Comments

PointsMetric
10All comments above the class, fields, and methods follow the JavaDoc comment style. Correct tags are added for the author, all method parameters and return types.
9All comments above the class, fields, and methods follow the JavaDoc comment style, and most of the comments contain and have correct all required tags. (At most 5 missing or misspelled tags.)
7Most comments above the class, fields, and methods follow the JavaDoc comment style, but a significant number of places are missing tags or the tags are misspelled so the comments are not formatted properly on the webpage.
5Most comments in JavaDoc style of /** but missing most tags OR has most tags but the comments do not start with /** and so do not appear on the webpage.
0No JavaDoc comments provided.

3. Correctness: 60 points

Performance correctness: 30 points

PointsMetric
30 All methods are done and correct.
27 Most of the methods are done and correct. Only minor errors are found.
24 Most of the methods are done and correct, but possibly significant errors in a few of the methods.
21 At least half of the equals, toString, compare, and loop methods of the database work across all of the classes.
18 Many significant errors in the methods, but at least one of the equals, toString, compare, or loop methods of the database work across all the classes.
9 The getter/setter methods of at least once class are correct.
0 Nothing works.

Good class hierarchy: 30 points

PointsMetric
30The class heirarchy is well organized and intuitive (every part of the hierarchy fits an "is-a" relation and all "is-a" relations in the project are included in the hierarchy), common methods moved as high up the hierarchy as appropriate, and polymorphism is used to simplify the code everywhere that is appropriate.
27The class heirarchy is well organized and intuitive (every part of the hierarchy fits an "is-a" relation, and most "is-a" relations in the project are included in the hierarchy) AND/OR polymorphism is used to simplify the code most of the places that are appropriate AND/OR most methods are moved as high up the hierarchy as appropriate.
24The class heirarchy has all of the "is-a" relations of the project, but includes a few places where an extending class does not fit the "is-a" relation AND/OR polymorphism is used in many of the places that are appropriate AND/OR many methods moved up the hierarchy but a significant number are unnecessarily duplicated between classes.
18A hierarchy is created but some of the extended classes do not follow the "is-a" relation and some of the "is-a" relations of the project are not in the hierarchy AND/OR polymorphism is used in some of the methods.
10Some attempt at creating a hierarchy but no polymorphism is used.
0No class hierarchy is created, no polymorphism is used.
